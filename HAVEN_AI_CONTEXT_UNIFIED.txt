WHAT HAVEN IS
━━━━━━━━━━━━━
HAVEN is a civic-tech platform for housing-insecure and underserved residents,
primarily serving the San José, CA area. It gives residents one place to:
  - Store and manage documents
  - Apply for government benefits and housing programs
  - Message their assigned caseworker
  - Chat with BB — an AI civic concierge
  - Access an offline-capable Survival Bible (emergency knowledge base)
  - Track submitted applications across multiple agencies

It simultaneously gives caseworkers, agencies, and administrators a full case
management system, workload tools, agency OAuth integrations, and reporting.

HAVEN is NOT a government website. It is a private platform that bridges
residents with government and non-profit agencies through a clean, dignified UX.
Every interaction is designed so residents feel relief, not bureaucracy.

USER ROLES
━━━━━━━━━━

RESIDENT
  The person seeking help. Can:
  - Create and track their own cases
  - Upload documents (ID, income, lease, medical)
  - Fill and submit 13 pre-built agency form templates
  - Chat with BB for guidance, form help, and crisis support
  - Message their caseworker per-case
  - Export their case as a full PDF packet for court or agency appointments
  - Access the Survival Bible and Camping/Crisis pages
  - View nearby resources on the interactive map
  Demo: resident@haven.demo / james@haven.demo / sarah@haven.demo (Demo2026!)

CASEWORKER
  Social worker, volunteer, or agency staff. Can:
  - See the full case queue (all cases, sorted by urgency)
  - Claim unassigned cases
  - Create and manage tasks per case
  - Send and receive messages per case
  - Verify uploaded documents
  - Use BB to autofill forms on behalf of residents
  - Submit applications to integrated agencies
  - Export full case PDF packets for inter-agency handoffs
  - See the workload heatmap (team capacity visualization)
  - Use the BB Browser Control panel to drive a real browser on agency sites
  Demo: caseworker@haven.demo (Demo2026!) — name: Alex Rivera

ADMIN
  Platform administrator. Can:
  - Everything a caseworker can do
  - Manage users (create, edit, deactivate)
  - Connect/disconnect agency OAuth integrations (Login.gov, VA, SSA, etc.)
  - View audit log of all platform actions
  - Access system analytics and dashboard
  - Manage agency integration catalog
  Demo: admin@haven.demo (Demo2026!) — name: Jordan Lee

ARCHITECT
  Superuser / platform owner.
  - Full access to everything including system configuration
  - Account: jawknee.rodriquez@gmail.com (Architect2026!)

BB — THE AI CIVIC CONCIERGE
━━━━━━━━━━━━━━━━━━━━━━━━━━━
BB is HAVEN's built-in AI assistant and the heart of the resident experience.
She is accessible via a floating chat bubble from any page on the platform.

Personality:
  - Warm, calm, dignity-first
  - Never judgmental, never bureaucratic, never robotic
  - Adapts tone to the user: anxious → calming; frustrated → validating; hopeful → momentum
  - Residents feel RELIEF. Caseworkers feel like BB saved them an hour of work.

BB's capabilities:
  1. Multi-turn context-aware conversation (remembers session history, up to 12 turns)
  2. Role-aware responses:
       Resident mode   — plain language, step-by-step, reduces overwhelm
       Caseworker mode — operational language, case IDs, action items, checklists
       Admin mode      — systems-oriented, metrics, anomalies, recommendations
  3. Form field analysis — paste any HTML form, BB extracts all fields with metadata
  4. Intelligent autofill — semantically maps resident profile + case intake data
     to form fields (e.g. "first_name", "dob", "household_size", "income")
  5. Application tracking — tracks submissions across agencies with timeline
  6. Crisis detection — automatically detects high-risk keywords and responds
     with appropriate hotlines + HAVEN actions
  7. BB Browser Control (advanced) — drives a headless Playwright browser to
     navigate and fill real agency websites on behalf of the resident
     (requires Playwright + chromium; available at /caseworker/bb-browser)
  8. Voice mode — mic input + speaker output via Web Speech API (Chrome/Edge)
     (Firefox not supported; buttons hidden automatically)
  9. Survival Bible expert — full knowledge of emergency skills (see below)

BB LLM engine:
  - Primary: local Ollama (llama3.2:3b default) — FREE, no API credits used
  - Fallback: Claude Sonnet 4.5 via Emergent key (only if Ollama is down)
  - Set EMERGENT_LLM_KEY= (blank) to fully disable Sonnet and all credit usage
  - /api/health reports which engine is active: "ollama_native" or "emergent_llm_fallback"

Crisis keywords and response levels:
  CRITICAL: "suicide", "kill myself", "hurt myself", "domestic violence",
            "being hit", "evicted today/tomorrow", "homeless tonight", "child in danger"
  HIGH:     "end it", "abuse", "no food", "kids hungry"
  → BB surfaces the relevant crisis hotline AND recommends a HAVEN action immediately

CORE FEATURES
━━━━━━━━━━━━━

CASE MANAGEMENT
  Cases are the central unit of work on HAVEN.
  - Each case belongs to one resident; one caseworker can be assigned
  - Caseworkers can self-assign via "Claim" on unassigned cases
  - Status lifecycle: new → enriched → routed → active → resolved → closed
  - Urgency score: 0–100 integer (higher = more urgent, shown as colored ring)
  - Categories: housing | food | health | benefits | employment | legal | crisis | general
  - Each case contains: title, description, category, urgency_score, intake_data dict
  - Attached to each case: tasks, messages, documents, form submissions
  - Case detail page (/caseworker/cases/:id) shows full case history
  - Residents see only their own cases; caseworkers see all cases

TASKS
  - Created by caseworkers on any case
  - Priority levels: low | medium | high | urgent
  - Statuses: open | in_progress | done
  - Optional due dates
  - Visible in case detail and workload heatmap

DOCUMENTS
  - Residents upload files through ResidentDocumentsPage
  - Document types: identity | income | residency | medical | other
  - Caseworkers can verify documents (marks as verified with their ID)
  - Max 8MB per file, stored as base64 in MongoDB (demo mode)
  - Documents index included in case packet PDF export

MESSAGING
  - Secure per-case messaging between resident and caseworker
  - Accessible via ResidentMessagesPage and CaseDetailPage
  - Threads are per-case, with read/unread tracking
  - Last 20 messages included in case packet PDF export

CASE PACKET PDF EXPORT
  Endpoint: GET /api/cases/{id}/packet.pdf
  One-click export for court appearances, appeals, and inter-agency handoffs.
  Auth: resident owner OR assigned caseworker OR admin → 200
        different resident → 403 | nonexistent case → 404
  Contents (8 sections, each with a navy HAVEN header bar):
    1. Cover page — case ID, title, category, status, urgency, resident, caseworker, timestamps
    2. Case summary — full description + intake_data key/value grid
    3. People — resident and caseworker contact cards
    4. Submitted applications — all form submissions for this case
    5. Documents index — filename, type, upload date, verified status
    6. Tasks & notes — title, priority, status, due date
    7. Message digest — last 20 messages, oldest to newest, sender identified
    8. Attestation footer — generation timestamp + HAVEN attribution
  Filename format: HAVEN_CasePacket_{case_id[:8]}.pdf

AGENCY INTEGRATIONS (12 agencies)
  The Agency Bridge connects HAVEN to real government systems.
  All run in simulated mode by default (deterministic mock responses + realistic confirmation IDs).
  Live OAuth mode activates when credentials are set via environment variables.

  Agency catalog:
    HUD_SEC8      HUD Section 8 Housing Choice Voucher    housing     federal
    SSA_SSI       SSA Supplemental Security Income        benefits    federal
    USDA_SNAP     USDA SNAP / CalFresh                    food        federal/state
    CMS_MEDICAID  CMS Medicaid / Medi-Cal                 health      federal/state
    VA_BENEFITS   VA Veterans Benefits Administration      benefits    federal
    IRS_EFILE     IRS VITA Free Tax Filing                benefits    federal
    DMV_ID        CA DMV REAL ID / State ID               identity    state
    DOL_UI        DOL Unemployment Insurance (EDD)        employment  federal/state
    HHS_TANF      HHS TANF / CalWORKs                     benefits    federal/state
    CPS_REPORT    County CPS Child Welfare Referral        family      county
    COURT_EFILE   Santa Clara Superior Court e-filing     legal       county
    WIC_CA        CA WIC Women, Infants & Children         health      federal/state

  Submission flow:
    1. Caseworker or resident fills form template
    2. POST /api/integrations/submit with integration_code + case_id + payload
    3. Adapter (simulated or live) returns confirmation_id + status + SLA date
    4. Submission stored with full timeline; mirrored to application_tracking
    5. POST /api/integrations/submissions/{id}/sync to check current status

  Live OAuth adapter families:
    logingov  — Login.gov OIDC + PKCE mandatory
                Used by: HUD, USDA, IRS, WIC, DOL, DMV, HHS, CMS
    va        — VA Lighthouse API with PKCE
    ssa       — SSA developer sandbox
    oauth2    — generic RFC 6749 authorization_code flow
    simulated — default when no credentials present

  To enable sandbox OAuth for an agency (minimum setup):
    OAUTH_ENV=sandbox
    HUD_SEC8_OAUTH_CLIENT_ID=your_login_gov_client_id
    (sandbox authorize/token URLs are built-in for all three families)

  OAuth security:
    - PKCE mandatory for Login.gov and VA (code_verifier stored server-side)
    - State tokens expire after 15 minutes (OAUTH_STATE_TTL_SECONDS=900)
    - State is single-use (deleted on success AND on expiry)
    - No client secrets in code — env vars only
    - Revoke via POST /api/integrations/{code}/oauth/disconnect

COMPLETE API SURFACE (prefix: /api)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Auth:
  POST  /auth/register                       create account (role: resident|caseworker|admin)
  POST  /auth/login                          returns { token, user }
  GET   /auth/me                             current user profile

Users:
  GET   /users                               list users (caseworker/admin, ?role=)
  PATCH /users/me                            update own name/phone

Cases:
  POST  /cases                               create case
  GET   /cases                               list (?status= ?category= ?assigned_to_me= ?caseworker_id=)
  GET   /cases/{id}                          detail + tasks + messages + documents
  PATCH /cases/{id}                          update (caseworker/admin)
  POST  /cases/{id}/claim                    caseworker self-assigns
  GET   /cases/{id}/packet.pdf               8-section PDF export

Tasks:
  POST  /tasks                               create task (caseworker/admin)
  GET   /tasks                               list (?case_id= ?status=)
  PATCH /tasks/{id}                          update
  DELETE /tasks/{id}                         delete

Messages:
  POST  /messages                            send message
  GET   /messages?case_id=                   thread for a case

Documents:
  POST  /documents                           upload file (multipart/form-data)
  GET   /documents?case_id=                  list documents
  PATCH /documents/{doc_id}/verify           verify document (caseworker/admin)

BB:
  POST  /bb/chat                             chat with BB → { reply, intent, session_id, timestamp }
  GET   /bb/intro                            role-aware greeting → { reply, role }
  GET   /bb/sessions/{session_id}/history    full chat history for a session
  POST  /bb/forms/analyze                    extract fields from HTML → { field_count, fields }
  POST  /bb/forms/autofill                   map profile + case data to form fields → { mapping, missing }
  POST  /bb/applications/track              track an application submission
  GET   /bb/applications/summary             all tracked applications for current user
  GET   /bb/applications/case/{case_id}      applications for a specific case
  GET   /bb/applications/status/{id}         single tracking record
  POST  /bb/browser/start                    start Playwright browser (returns screenshot)
  POST  /bb/browser/action                   navigate|fill|click|type|submit|screenshot|
                                              extract|scroll|back|forward|autofill_all
  POST  /bb/browser/stop                     close browser session
  GET   /bb/browser/sessions                 list active browser sessions
  WS    /bb/browser/stream/{session_id}      live screenshot stream (?token=JWT)

Form Templates:
  GET   /form-templates                      list all 13 (no fields, summary only)
  GET   /form-templates/{id}                 full template with all fields
  POST  /form-templates/{id}/autofill        deterministic autofill (no LLM cost)
  POST  /form-templates/{id}/submit          submit filled form → { id, template_name, submitted_at }
  GET   /form-templates/submissions/mine     all my submissions
  GET   /form-templates/submissions/{id}     single submission
  GET   /form-templates/submissions/{id}/pdf HAVEN-branded PDF of single submission

Agency Integrations:
  GET   /integrations                        full agency catalog
  GET   /integrations/{code}                 single agency
  PATCH /integrations/{code}/toggle          connect/disconnect (admin)
  POST  /integrations                        create custom integration (admin)
  DELETE /integrations/{code}               delete integration (admin)
  POST  /integrations/submit                 submit application to agency
  GET   /integrations/submissions            all submissions (?case_id= ?integration_code= ?status=)
  GET   /integrations/submissions/{id}       single submission with timeline
  POST  /integrations/submissions/{id}/sync  check status from agency (live or simulated progression)
  GET   /integrations/{code}/oauth/meta      mode + adapter_family + authorized status
  GET   /integrations/{code}/oauth/start     begin OAuth flow (admin) → { authorize_url, adapter_family }
  GET   /integrations/oauth/callback         OAuth redirect handler (PKCE verifier + state expiry enforced)
  POST  /integrations/{code}/oauth/disconnect revoke tokens (admin)
  GET   /integrations/_/stats               submission counts, live_configured, by_category

Resources:
  GET   /resources                           all map resources

Notifications:
  GET   /notifications                       in-app notifications for current user

Forms (custom builder):
  POST  /forms                               create custom form
  GET   /forms                               list published forms
  GET   /forms/{id}                          form detail
  POST  /forms/{id}/submit                   submit to custom form

Analytics:
  GET   /analytics/admin                     system analytics (admin/architect)
  GET   /analytics/workload-heatmap          caseworker load matrix (caseworker/admin)

Curriculum:
  GET   /curriculum/modules                  role-aware module list
  GET   /curriculum/modules/{id}             module detail (answers stripped)
  POST  /curriculum/submit-quiz              quiz submission (80% pass threshold)
  GET   /curriculum/progress                 current user's progress

Health:
  GET   /health                              full health: API + DB + BB engine status
  GET   /health/live                         liveness probe → { alive: true }
  GET   /health/ready                        readiness probe (DB ping) → { ready, db }

KNOWN LIMITATIONS
━━━━━━━━━━━━━━━━━
- BB Browser Control requires Playwright + chromium (not in base Docker image)
  Enable: pip install playwright && python -m playwright install chromium
- Firefox lacks Web Speech API — mic/speaker buttons hidden automatically
- PDF export requires reportlab==5.0.0 in requirements.txt
- Refresh token rotation for Login.gov/VA not yet implemented
- i18n scaffold present (LanguageSwitcher component) but not fully activated
- /api/health returns "degraded" when on Emergent LLM fallback — known label quirk, not a regression
- Signed URLs for PDF downloads not yet implemented (currently JWT + ownership check only)

DEPLOYMENT
━━━━━━━━━━
Services (docker-compose.yml):
  nginx     ports 80/443 — SSL termination, HTTP→HTTPS, SPA routing, API proxy
  frontend  port 3000    — React/Vite static build served via `serve`
  backend   port 8000    — FastAPI, 2 uvicorn workers
  mongo     port 27017   — MongoDB 7.0, volume: mongo-data
  ollama    port 11434   — pulls llama3.2:3b on first start (~2GB), volume: ollama-models

Domain routing:
  homeishaven.cloud         → main site (HTTPS)
  www.homeishaven.cloud     → main site (HTTPS)
  home-is-haven-com         → 301 → homeishaven.cloud
  www.home-is-haven-com     → 301 → homeishaven.cloud

Commands:
  Start:   docker compose up -d --pull always
  Logs:    docker compose logs -f backend
  Rebuild: docker compose build --no-cache
  Seed DB: docker compose exec backend python seed.py
